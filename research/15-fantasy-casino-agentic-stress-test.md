# Fantasy Casino as Agentic Stress Test

## Working title

**Fantasy Casino as Agentic Stress Test**

## Core thesis

This project is not primarily about building a casino. It is about using a casino-shaped domain as a hard, realistic stress test for autonomous local software development, safe agent orchestration, and public engineering leadership growth.

The casino is valuable because it forces the system to deal with money, auditability, compliance, risk, telemetry, product complexity, and high-stakes correctness.

## What the project should prove

The experiment should prove that a personal laptop can support a disciplined local agent loop that:

- reads and understands a large repo in chunks
- plans meaningful work instead of tiny prompts
- patches code safely inside a sandbox
- never deletes files directly
- runs verification and learns from failures
- reports progress through Telegram every 5 minutes while active
- sends screenshots when visual evidence helps
- resumes cleanly after interruption
- uses local models as the default backbone
- escalates to stronger external models only when justified

## What the project should become

The project should evolve into three things at once:

1. A real engineering experiment.
2. A product/runtime blueprint for a casino-like domain.
3. A public proof of your engineering maturity, judgment, and operating model thinking.

## Current stack decision

The experiment should use **Ruby on Rails** as the primary application stack.

That keeps the project aligned with your current working style and avoids unnecessary novelty.

The intent is to build something working, tested, and safe first. After that, specific pieces may be translated to other languages later if there is a clear reason to do so.

## Non-negotiable principles

### 1. Local-first, not local-only

The system should default to local compute, but it should not be ideological. External models are allowed when they improve quality, speed, or reliability.

### 2. Safety before autonomy

Autonomy is earned through evidence. Sensitive changes always require approval.

### 3. No direct deletion

If something must disappear from active use, move it to quarantine instead.

### 4. Chunked execution

The agent should work in meaningful chunks that can be explained, reviewed, and verified.

### 5. Evidence over claims

Every meaningful step should leave a trace:

- diff
- log
- screenshot
- test result
- short summary

## Public narrative

The public story should be:

> I am building a safe, observable, local-first autonomous engineering lab using a realistic casino domain as a stress test for agentic development.

That story is stronger than “I am automating a casino”.

## Success signals

The project is moving in the right direction if:

- the agent can keep working in bounded chunks
- the heartbeat is useful and honest
- the Telegram updates make the work understandable
- the control plane stays visible
- local models handle a large share of routine tasks
- the human remains in charge of risk
- the system produces reusable knowledge, not just code

## What not to forget

- This is a growth platform.
- This is a safety experiment.
- This is a visibility engine.
- This is a product architecture test.
- This is a local-AI benchmark on real work.

If the project loses those five properties, it has drifted off mission.
